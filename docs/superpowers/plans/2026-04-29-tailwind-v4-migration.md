# Tailwind CSS v4 Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate the YAS frontend from Tailwind v3.4 to v4.x using the modern v4 idioms (CSS-first `@theme`, `@tailwindcss/vite` plugin, `tw-animate-css`, freshly regenerated shadcn primitives), keeping existing HSL CSS variables.

**Architecture:** Five-task migration on a single branch, single PR. Task 1 commits the official `@tailwindcss/upgrade` codemod output verbatim so reviewers can isolate "the tool did this" from "we did this." Task 2 swaps to the Vite plugin and the CSS-first config. Task 3 re-shadcns the 10 primitives. Task 4 is the audit / no-change verification pass. Task 5 runs all gates including the e2e and a manual visual click-through, then pushes and opens the PR.

**Tech Stack:** React 19, Vite 8, TypeScript 5.9, Tailwind v4.x, `@tailwindcss/vite`, `tw-animate-css`, shadcn/ui (slate base, cssVariables: true, default style).

**Spec:** `docs/superpowers/specs/2026-04-28-tailwind-v4-migration-design.md`

**Project conventions to maintain:**
- All deps pinned to exact patch (no `^`/`~`) in `frontend/package.json`. Use `npm install --save-exact` for every new dep.
- All commits must be signed via the 1Password SSH agent. Set `SSH_AUTH_SOCK="$HOME/Library/Group Containers/2BUA8C4S2C.com.1password/t/agent.sock"` before each `git commit` in fresh shells (subagents do NOT inherit it). Verify with `git cat-file commit HEAD | grep -q '^gpgsig '` after every commit.
- Branch off `main` into `frontend/tailwind-v4`. Do NOT commit to `main` directly; the user will merge the PR after CI passes.
- Frontend gates: `npm run typecheck`, `npm run lint`, `npm run test -- --run`, `npm run build`, plus `./scripts/e2e_phase5a.sh` from the repo root.
- ESLint stays on v8.57 with `.eslintrc.cjs` (legacy). Do NOT migrate to flat config.

- Backend code, alembic, tests/integration — Tailwind migration is frontend-only.

## Notes for the implementer

- **Single PR, single branch**. Tailwind v4 is a global build-tool change; partial states cannot ship to main.
- **No new runtime dependencies**. `tw-animate-css` is a CSS-only build-time package.
- **The codemod's output is rarely perfect**. Expect to do real work in Task 2 to get to the desired shape. Don't fight the codemod; just commit its output and then layer manual fixes.
- **Pin every new dep with `--save-exact`**. The project's existing pin policy is enforced by Renovate's `rangeStrategy: pin`; any `^` or `~` introduced will be auto-corrected later but is still wrong on first commit.
- **HSL stays.** Do not touch the OKLCH-vs-HSL question. The new shadcn primitives reference `bg-background`, `text-foreground`, etc. — names that resolve via the `@theme inline` block to `hsl(var(--background))`, etc. There is no need to rewrite color literals.
- **If `npm run dev` works but `npm run build` fails (or vice versa)**, that's the v4 build difference between dev (esbuild via Vite) and prod (Lightning CSS). Trust the prod build — read the Lightning CSS error and fix the source.
- **The existing `TopBar.test.tsx` and `AlertsSection.test.tsx`** render shadcn primitives via happy-dom. They assert on text content, not on CSS values. They should be unaffected by the migration unless a primitive's DOM structure changed materially (rare).
